《疯狂SpringCould》 Eureka

一、简介

  • 提供基于REST的服务，在集群中用于服务管理

  • 客户端组件实现了负载均衡，方便业务组件集群部署

  • Eureka主要维护这些服务的列表并自动检查他们的状态

  • 无论是服务器还是客户端，都可部署多个实例

1.服务器端

对于注册到服务器端的服务组件，Eureka服务器并没有提供后台的存储，这些注册服务实例被保存在内存的注册中心，它们通过心跳来保持最新状态，操作都在内存中完成。

客户端存在相同机制，也在内存中保存了注册表信息。这样提升了Eureka组件的性能，每次服务的请求都不必经过服务器端的注册中心

四、健康检查

默认情况下，Eureka客户端每30s会发送一次心跳给服务器端。但在实际环境中，可能出现这种情况：客户端表面上可以发送心跳，但服务不可用。如：服务提供者需要访问数据库，但数据库无法访问，但表面上可以响应客户端；或访问三方服务失效。因此应该告诉服务器当前客户的状态，使用Eureka健康检查控制器

1.服务提供者检查

@Component

    @Override

 */

    @Autowired
    private MyHealthindicator indicator;

Eureka中会启动一个定时器，定时刷新本地实例的信息，并且执行处理器中的getStatus()方法，再将服务实例的状态更新到服务器中。默认30s执行一次

可以更改配置加快执行速度：eureka.client.instance-info-replication-interval-seconds=10

五、Eureka常用配置

1.心跳检测配置

# （客户端）客户端实例会向服务端发送周期性的心跳，默认30s发送一次
eureka.instance.lease-renewal-interval-in-seconds=30
# （客户端）服务端接收心跳请求，若90s内没有收到心跳，就会将实例从注册表删除
eureka.instance.lease-expiration-duration-in-seconds=90


# （服务端）清理注册表的定时器，默认60s执行一次
eureka.server.eviction-interval-timer-in-ms=6000
# （服务端）如果开启自我保护模式，实例不会被清除
eureka.server.enable-self-preservation=false

2.注册表（可用的服务列表）抓取间隔

# （客户端）每隔30s去服务器抓取注册表，并将服务器的注册表保存到本地缓存
eureka.client.registry-fetch-interval-seconds=30

3.配置与使用元数据

框架自带的元数据，如实例id、主机名、ip地址等。如果需要自定义元数据提供给客户端使用

# （服务端）配置元数据
eureka.instance.metadata-map.name=xxx
eureka.instance.metadata-map.age=18

配置后，元数据会保存在服务器注册表中，并且使用简单的方式与客户端进行共享

@Autowired
private DiscoveryClient discoveryClient;


@RequestMapping(“/test”)
public String test(){
    // 获取服务实例
    List<ServiceInstance> ins = discoveryClient.getInstances(“server001”);
    for(ServiceInstance si: ins){
        // 获取元数据
        System.out.println(si.getMetadata().get(“name”));;
    }
    return “”;
}

4.自我保护模式

Eureka的主界面看到的红色提醒。

客户端心跳的失败率超过一定比例，服务会将这些事例保护起来，并不会马上将其从注册表中删除。此时客户端可能会拿到无法使用的实例

可以配置关闭自我保护模式
